# 会话复用方法性能对比详解

## 性能提升百分比说明

### 计算公式

```
性能提升% = (第一轮耗时 - 第二轮耗时) / 第一轮耗时 × 100%
```

- **正数 (+%)**: 第二轮比第一轮快（性能提升）
- **负数 (-%)**: 第二轮比第一轮慢（性能下降）

## 详细对比表

| 方法 | 第一轮耗时 | 第二轮耗时 | 节省时间 | 性能提升 | 说明 |
|------|-----------|-----------|---------|---------|------|
| **continue** | 10,301ms | 6,006ms | -4,295ms | **+42%** | 自动复用最近会话，无需跟踪 ID |
| **resume** | 10,806ms | 5,096ms | -5,710ms | **+53%** | 精确恢复指定会话，性能最佳 |
| **forkSession** | 12,468ms | 27,674ms | +15,206ms | **-122%** | 创建分支会话，需要复制历史 |
| **单次 query 多轮** | 7,139ms* | 3,944ms/轮 | -3,195ms | **+45%** | 一次启动完成多轮，最快方案 |

> *单次 query 多轮的第一轮使用创建新会话的基线耗时 (7,139ms) 作为对比基准

## 绝对耗时对比

### 场景：需要执行 10 轮对话

| 方法 | 总耗时估算 | 说明 |
|------|-----------|------|
| 每次创建新会话 | ~71,390ms | 10 × 7,139ms，最慢 ❌ |
| continue | ~64,365ms | 10,301 + 9×6,006，提升 10% |
| resume | ~56,670ms | 10,806 + 9×5,096，提升 21% ✅ |
| 单次 query 多轮 | ~39,440ms | 10 × 3,944，提升 45% ✨✨✨ |

> 结论：对于多轮对话，**单次 query 多轮**是最优方案

### 场景：跨时间段恢复会话

| 方法 | 适用性 | 说明 |
|------|-------|------|
| continue | ❌ 不适用 | 只能复用最近会话 |
| resume | ✅ 适用 | 精确恢复指定会话 |
| forkSession | ✅ 适用 | 探索不同方案 |

## 选择建议

### 1. 交互式对话（实时聊天）
```
推荐: continue
理由: 无需管理 session ID，自动复用最近会话
性能: 提升 42%
```

### 2. 多用户系统（SaaS 应用）
```
推荐: resume
理由: 每个用户独立的 session ID，精确恢复
性能: 提升 53%（最佳）
```

### 3. 批量处理（自动化脚本）
```
推荐: 单次 query 多轮
理由: 一次启动完成所有任务，最快
性能: 提升 45%（绝对耗时最低）
```

### 4. 探索性开发（尝试不同方案）
```
推荐: forkSession
理由: 保留原会话，创建新分支探索
性能: 下降 122%（可接受的代价）
```

## 总结

- **性能提升**是指相对于"每次创建新会话"的改进程度
- **resume** 性能提升最大（+53%），但需要管理 session ID
- **continue** 使用最简单，性能提升也不错（+42%）
- **单次 query 多轮** 绝对耗时最低，适合批量处理
- **forkSession** 会变慢（-122%），但用于探索性场景是值得的
