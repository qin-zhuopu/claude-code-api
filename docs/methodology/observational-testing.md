# 观察性测试方法论

一种通过"调研 → 观察 → 断言 → 沉淀"循环来理解黑盒系统行为的工程方法。

## 适用场景

- 对接第三方 SDK / API，文档不完整或行为不明确
- 需要理解系统内部机制但无法阅读源码
- 验证文档描述与实际行为是否一致
- 探索参数组合的交互效应

## 核心流程

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  1. 调研     │ ──→ │  2. 观察     │ ──→ │  3. 断言     │ ──→ │  4. 沉淀     │
│  收集输入    │     │  录得数据    │     │  验证假设    │     │  形成文档    │
└──────────────┘     └──────────────┘     └──────────────┘     └──────────────┘
       ↑                                                              │
       └──────────────────── 发现新问题，回到调研 ←────────────────────┘
```

### 第一步：调研（收集输入）

**目标**：建立对系统行为的初步假设。

输入来源：
- 官方文档（API 文档、SDK 类型定义）
- 静态代码分析（类型签名、注释、示例）
- 已有日志/请求体（历史运行数据）
- 社区讨论、changelog

方法：
1. 阅读类型定义，列出所有可配置参数
2. 阅读文档中的行为描述，形成初步假设
3. 对比已有日志，发现文档未提及的行为
4. 列出需要验证的问题清单

### 第二步：观察（录得数据）

**目标**：通过实际运行获取真实数据。

方法：
1. 编写最小化测试用例，每次只变一个变量（变量控制法）
2. 通过日志/拦截机制捕获完整的请求/响应数据
3. 将数据持久化到文件系统，便于事后分析
4. 用脚本提取结构摘要，避免人工阅读大文件

关键原则：
- **变量控制法**：每个用例只改变一个变量，其余保持不变
- **日志隔离**：每个用例的日志写入独立目录，按编号命名
- **数据持久化**：原始数据 + 格式化数据都保留
- **结构提取**：先看结构（key tree、block count、char count），再看内容

### 第三步：断言（验证假设）

**目标**：将观察到的行为固化为可重复验证的断言。

方法：
1. 第一轮：宽松断言（只记录观察，不做硬断言）
2. 根据实际数据修正假设
3. 第二轮：精确断言（基于已确认的行为）
4. 处理意外发现（修正断言 or 记录为新发现）

断言策略：
- 对确定的行为：硬断言（`expect(x).toBe(y)`）
- 对不确定的行为：先用 `console.error` 记录，确认后再加断言
- 对交互效应：用对比断言（`expect(a).toBeGreaterThan(b)`）

### 第四步：沉淀（形成文档）

**目标**：将验证过的知识转化为可复用的文档。

文档结构：
1. 结论先行（一句话总结核心发现）
2. 实验矩阵（表格形式展示所有用例和结果）
3. 各参数独立作用（逐个解释）
4. 交互效应（组合行为）
5. 实际应用建议（推荐配置）
6. 测试文件引用（可复现）

## 变量控制法详解

### 设计原则

1. **确定基准**：选择一个已知可工作的配置作为对照组
2. **单变量翻转**：每次只改变一个参数
3. **覆盖边界**：测试每个参数的所有有意义取值
4. **观察交互**：在单变量测试完成后，测试关键组合

### 用例编号规范

```
case-1  基准（对照组）
case-2  变量A取值1
case-3  变量A取值2
case-4  变量B取值1
...
case-N  交互效应（A+B组合）
```

### 日志目录规范

```
tmp/<test-suite-name>/
├── case-1-baseline/
│   └── <timestamp>/
│       ├── <uuid>.request.json
│       ├── <uuid>.request.pretty.json
│       ├── <uuid>.response.json
│       └── <uuid>.response.pretty.json
├── case-2-xxx/
└── case-3-yyy/
```

## 迭代优化

### 第一轮：探索性测试

- 断言宽松，主要目的是录得数据
- 用 `console.error` 输出关键指标
- 快速跑通所有用例

### 第二轮：精确化

- 根据第一轮数据修正断言
- 处理意外发现（如 "自定义 string 不会替换默认 prompt"）
- 补充遗漏的用例

### 第三轮：文档化

- 将所有发现整理为结构化文档
- 标注哪些是确认的行为，哪些是推测
- 给出实际应用建议

## 工具链

| 环节 | 工具 |
|------|------|
| 数据捕获 | OTEL_LOG_RAW_API_BODIES、请求拦截、日志文件 |
| 结构提取 | 自定义脚本（pretty-json-keys.js、compare-requests.js） |
| 测试框架 | vitest（支持超时、并行、过滤） |
| 数据格式化 | JSON.stringify(obj, null, 2) |
| 文档格式 | Markdown 表格 + 代码块 |

## 反模式

| 反模式 | 正确做法 |
|--------|---------|
| 一次改多个变量 | 每次只改一个，逐步排列组合 |
| 只看文档不验证 | 文档是假设，实验是验证 |
| 断言写死后不改 | 第一轮宽松，根据数据迭代 |
| 日志混在一起 | 每个用例独立目录 |
| 只记结论不记数据 | 原始数据和格式化数据都保留 |
| 大文件人工阅读 | 先用脚本提取结构摘要 |
